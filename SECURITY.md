# Security Policy

lkhu stores your Claude Code memories as latent vectors on your own machine. Everything — embedding, storage, recall, consolidation — runs locally. There is no cloud backend, no API key, and no telemetry. This document explains exactly what lkhu touches, what it never does, and how to report a vulnerability if you find one.

## Supported versions

| Version | Supported |
|---------|-----------|
| 0.1.x   | ✅ Yes |
| < 0.1   | ❌ No — please upgrade |

Only the latest 0.1.x release receives security fixes. lkhu is pre-1.0, so upgrading is the expected remediation path.

## Reporting a vulnerability

Please report vulnerabilities **privately** — not via a public issue.

- **Preferred:** [GitHub Security Advisories](https://github.com/iddk0321/lkhu/security/advisories/new) (repository → Security → Report a vulnerability)
- **Email:** developer.donguk@gmail.com

You can expect an acknowledgment within 7 days. Please include a reproduction if you can, and give us reasonable time to ship a fix before public disclosure.

## What lkhu touches

lkhu's footprint is deliberately small. Here is the complete surface.

### Local files

All data lives in a per-user directory resolved via `platformdirs` (override with the `LKHU_DATA` environment variable):

| OS | Data directory |
|----|----------------|
| macOS | `~/Library/Application Support/lkhu` |
| Linux | `~/.local/share/lkhu` |
| Windows | `%LOCALAPPDATA%\lkhu` |

Inside it: `memories.db` (SQLite), `vectors.faiss`, `codebook.npy` (plus `codebook.backup.npy`, each with a `.meta.json` checksum sidecar), `short_term.npy`, `audit/` (append-only JSONL logs, one file per day), `backups/`, `logs/`, and `stats.json`. A third codebook backup is written to `~/Documents/lkhu_codebook.backup.npy`.

Outside the data directory (and the Documents codebook backup above), lkhu edits config files only when you explicitly ask it to: `lkhu install` (or `lkhu init`) registers an MCP server entry in the **Claude Desktop** config (`claude_desktop_config.json`), preserving all other entries, and `lkhu uninstall` removes lkhu's own entries — including any legacy direct Claude Code MCP/hook registrations. The optional `lkhu install-hooks` command adds lkhu's hooks to the Claude Code user settings (`~/.claude/settings.json`); `lkhu uninstall-hooks` reverts only those. The recommended Claude Code integration goes through the official plugin system and edits no config files. (`lkhu export` writes a plaintext file wherever you point it — see below.)

### Local HTTP — two loopback services

- **Ollama.** Embeddings are generated by your locally installed Ollama server running `snowflake-arctic-embed2` (the default embedder; `bge-m3` and any other 1024-dim model remain selectable in your user config). lkhu talks to it over local HTTP via the official `ollama` Python client. Your text is embedded on your machine, never sent to a remote embedding API.
- **The lkhu daemon.** A stdlib `ThreadingHTTPServer` that owns the memory engine. It binds to **`127.0.0.1`** by default, on port **`37700 + (your uid % 100)`** — e.g. `37701` for the typical first macOS user account. Hooks, the MCP server, and the CLI are all thin HTTP clients of this daemon, and the dashboard is served from the same address (e.g. `http://127.0.0.1:37701/`). Host and port can be overridden with `LKHU_DAEMON_HOST` and `LKHU_DAEMON_PORT`.

### Network

The memory pipeline makes **zero external network calls**. Save, recall, consolidation, decay, and cleansing are pure local vector math — no LLM API, no remote vector DB, no usage analytics. Even the decoder's optional Tier-3 "LLM fallback" slot is not wired to any model in the shipped configuration: it degrades to a plain text excerpt, so no recall path ever calls an LLM.

The only network activity in lkhu's orbit is Ollama downloading the embedding model the first time you run `ollama pull snowflake-arctic-embed2`. That is a one-time model download performed by Ollama itself — lkhu never downloads anything; if the model is missing, `lkhu install` and `lkhu doctor` only print a hint telling you to pull it.

## What's stored in plaintext

Memories are stored primarily as 1024-dim vectors, but lkhu keeps a natural-language **shadow copy** (`audit_text`) for your visibility and debugging:

- `memories.db` and the `audit/` JSONL logs contain conversation excerpts in plaintext.
- `lkhu export` produces a plaintext JSONL file. Treat exports and `backups/` like any file containing conversation history before sharing or syncing them.
- Data at rest is **not encrypted** by lkhu. It relies on your OS user permissions and full-disk encryption, the same as your shell history or editor state.

To keep something out of memory entirely, wrap it in `<private>...</private>` in your prompt — the hooks strip private blocks before anything is saved.

## Known limitations

Be aware of these when threat-modeling your setup:

- **The daemon API has no authentication.** It is loopback-only by default, so the boundary is "processes on your machine running as any local user." Any local process can read or write memories through it. Do **not** set `LKHU_DAEMON_HOST` to a non-loopback address — that would expose your memories, unauthenticated, to your network.
- **The codebook is integrity-checked, not secret.** `codebook.npy` is verified on load against a SHA-256 checksum stored in its `.meta.json` sidecar, and saving refuses to overwrite an existing codebook. But anyone with read access to your data directory can interpret your vectors — file permissions are the protection layer.
- **Hooks fail open.** `lkhu hook` is designed to never block your Claude Code session: any internal error returns a pass-through result. This is a deliberate availability-over-strictness trade-off.
