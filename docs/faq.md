# FAQ

Quick answers to the questions people ask most. If you're brand new, start with the [docs index](index.md) and the install guide for your OS ([macOS](installation/macos.md) / [Linux](installation/linux.md) / [Windows](installation/windows.md)).

## Basics

### How is this different from claude-mem?

claude-mem stores memories as **natural-language summaries** generated by Claude API calls and searches them with full-text + vector search — that makes it excellent at retrieving exact past content with citations. lkhu stores memories as **1024-dim latent vectors** ("scents") computed entirely on your machine, with zero LLM calls in the save/recall/consolidate/decay/cleanse pipeline. The trade-off: claude-mem gives you richer, exact observation detail; lkhu gives you full local privacy, no API cost, and a biomimetic lifecycle where memories strengthen and fade. See the [full comparison](comparison.md) for an honest side-by-side.

### Does it really save tokens compared to RAG?

Conventional RAG injects retrieved document chunks as raw text into the prompt, and summary-based memory systems spend API tokens compressing every session. lkhu's pipeline uses no LLM at all for storing, consolidating, and cleansing, and decoding recalls is 3-tier: roughly 70% end at a Tier-1 audit excerpt, ~25% at a Tier-2 key-unbind probe, and under 5% reach the Tier-3 LLM fallback — which is capped at 80 tokens. Injected memory lines are truncated at 200 characters each. The design target is 1/5–1/10 the token cost of conventional natural-language RAG.

## Privacy & dependencies

### Does my data leave my machine?

No. Embeddings are computed by Ollama running `snowflake-arctic-embed2` locally, memories live in SQLite + FAISS files under your local data directory, and the lkhu daemon binds to `127.0.0.1` (localhost) by default. By default the decoder's Tier-3 fallback is a non-LLM extractive excerpt, so the memory pipeline makes zero external API calls — no API key, no API bill.

### Why are Ollama and snowflake-arctic-embed2 required?

Every memory is a vector in `snowflake-arctic-embed2`'s 1024-dimensional multilingual embedding space, and all of lkhu's math (binding, bundling, similarity search) assumes that one consistent space. Ollama is the local runtime that serves the model, which is what keeps everything on your machine. Install Ollama from [ollama.com](https://ollama.com), then:

```bash
ollama pull snowflake-arctic-embed2
```

`bge-m3` was the previous default and still works if you set it explicitly (see below), but `snowflake-arctic-embed2` now wins lkhu's own eval on retrieval and cross-lingual recall.

### How do I change the embedding model?

Set `encoder.model` in your user `config.yaml` (in the OS config dir, e.g. `~/.config/lkhu/config.yaml` on Linux), pull the new model with `ollama pull <model>`, then re-encode your existing memories:

```bash
lkhu reembed --yes
```

Existing vectors were computed by the old model and live in a different space, so recall breaks until you re-embed; `reembed` rebuilds every scent with the new model while preserving each memory's text, strength, and metadata. Stop the daemon / Claude Code first, since they hold the same store. The codebook is keyed to the **dimension** (1024), not the model — so any other 1024-dim model keeps your codebook and only needs a `reembed`.

### What if Ollama is down?

Nothing breaks. The hooks are designed to **never block your work**: any error — including an unreachable Ollama — makes them return a pass-through response, so Claude Code continues normally while memory save/recall silently no-ops. Your existing memories are safe on disk; once Ollama is back, everything resumes. Run `lkhu doctor` to diagnose — it checks the Ollama server, codebook integrity, MCP registration, hooks, and the daemon.

### Does it work in my language?

Yes. `snowflake-arctic-embed2` is a multilingual embedding model, and lkhu contains no per-language keyword lists anywhere — even the noise filter that decides what's worth saving is length-based, not word-based. Memories stored in one language are recalled by queries in another, because both land in the same semantic vector space.

```bash
# Example (Korean input, English query — multilingual recall):
lkhu remember "이 프로젝트는 탭 대신 스페이스 4칸을 쓴다"
lkhu recall "indentation style"
```

## Day-to-day use

### How do I see what it remembers?

Run `lkhu dashboard` — it starts the daemon if needed and opens a live web dashboard showing every memory with its strength, kind, recall count, and age. For a quick terminal summary, `lkhu status` prints totals and distributions, and `lkhu recall "some query"` shows you exactly what a search returns. For the full raw record, `lkhu export` writes the human-readable audit log as JSONL.

### How do I delete a memory? Or everything?

Three levels, from surgical to scorched-earth:

```bash
lkhu forget "the old database password" --confirm   # archive matching memories (audit preserved)
lkhu reset --confirm                                 # delete ALL data, including the codebook
lkhu uninstall                                       # remove integrations — data and codebook are KEPT
```

`forget` archives rather than deletes, so the audit text stays visible. `uninstall` only unregisters lkhu from Claude Desktop (run `claude plugin uninstall lkhu@lkhu` for the Claude Code side); your memories survive a reinstall. Only `reset --confirm` is irreversible.

### Where is my data stored?

In your OS's standard app-data location via `platformdirs`: `~/Library/Application Support/lkhu` on macOS, `~/.local/share/lkhu` on Linux, `%LOCALAPPDATA%\lkhu` on Windows. You can override it with the `LKHU_DATA` environment variable. See [storage.md](storage.md) for what each file is.

### Can I move my memories to another machine?

Yes — `lkhu export` on the old machine, `lkhu import <file>` on the new one. Import re-encodes each record's audit text with the new machine's own codebook, so the result is fully consistent locally even though raw vectors aren't portable across codebooks. Strength and recall history are not preserved; imported records start fresh.

## Memory lifecycle

### Why do old memories fade? Is that a bug?

It's the point. Like the Ebbinghaus forgetting curve, every memory's strength decays by ×0.99 daily, while each recall reinforces it by ×1.05 (Hebbian learning, capped at 1.5). Memories weaker than 0.1 and older than 30 days get archived during the weekly cleanse — never hard-deleted, so the audit text remains. Without forgetting, recall fills up with stale noise; decay is what keeps the relevant stuff on top. See [neuroscience.md](neuroscience.md) for the biology behind it.

### Is the codebook really unrecoverable if lost?

Yes — the codebook is the key dictionary that all structural binding is built on, and it cannot be regenerated identically (its seed is random and the code refuses to overwrite an existing one). That's why `lkhu install` creates it with a triple backup: `codebook.npy` and `codebook.backup.npy` in the data directory, plus `lkhu_codebook.backup.npy` in your Documents folder. Run `lkhu backup` periodically for timestamped copies of the codebook and database. Worst case, the audit shadow log still exists: `lkhu export` + `lkhu import` rebuilds your memories under a fresh codebook, at the cost of strengths and history.

## Under the hood

### Why numpy instead of torchhd?

HRR's bind/unbind operations are circular convolution and correlation, which are mathematically identical to multiplication in FFT space — so a pure numpy implementation produces the same results. torch's and faiss's OpenMP runtimes also clash in a multithreaded server (OMP error #179), and dropping torch makes the install roughly 1 GB lighter. The math is walked through in [vsa-explained.md](vsa-explained.md).
